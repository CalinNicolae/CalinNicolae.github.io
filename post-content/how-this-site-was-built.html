The Decision to Build from Scratch

When it came time to put together a portfolio for the Professional Networking module at Howest, the obvious shortcut would have been WordPress, Wix, or a pre-built GitHub Pages theme. I chose not to go that route. For a cybersecurity and computer science student, handing control of your public presence to a third-party CMS feels like a contradiction. More practically, a hand-built site demonstrates exactly the kind of skills that make this portfolio worth looking at.

This post documents how the site was built, how it works, and how to add new content to it — satisfying blocking factor 1c from the Howest Professional Networking module, which requires an explanation of the site's construction.

The Stack

The entire site runs on three technologies and zero dependencies:

  • HTML5 — semantic page structure, accessible markup
  • CSS3 — design system built entirely with custom properties, no preprocessor
  • Vanilla JavaScript (ES6+) — modules, fetch API, IntersectionObserver, dynamic rendering

Fonts are loaded from Google Fonts (JetBrains Mono and Syne) via a single @import in the stylesheet. There is no build step, no node_modules, no bundler. Opening index.html in a browser is all it takes.

How the Design System Works

Every colour, spacing value, and font is defined as a CSS custom property in :root inside style.css. Changing the primary neon accent from cyan to any other colour means editing one line. Utility classes like .neon-glow, .glitch, and .tag are defined once and reused across pages.

How the Blog System Works

The blog is driven by a single JSON file — data/posts.json — which acts as the data layer. Each entry holds the post's metadata: title, date, category, tags, excerpt, cover image path, and a pointer to the content file.

Post content lives in post-content/ as small HTML fragments — no <html>, <head>, or <body> tags. When you navigate to post.html?id=event-01, the page reads the URL parameter, fetches the matching entry from posts.json, renders all the metadata into the header, then fetches post-content/event-01.html and injects it into the article body. No server, no database — just fetch().

Adding a New Post

Three steps:

  1. Add a new JSON object to data/posts.json with all required fields.
  2. Create post-content/your-slug.html with the article body HTML.
  3. Drop any images into assets/images/ and update the paths in the JSON entry.

Push to the main branch and GitHub Pages deploys automatically within seconds.

Tools and Workflow

The site was built using Visual Studio Code, Git for version control, and browser developer tools for layout work. AI tools were used as a research and brainstorming aid throughout — to look up CSS behaviour, explore approaches to the dynamic post loading, and reason through animation logic. This is the same way a developer might use Stack Overflow, MDN, or official documentation: as a resource to accelerate specific problems, not as a replacement for writing and understanding the code. Every line in this repository was read, reasoned about, and deliberately placed.

Hosting

The site is hosted on GitHub Pages, free, reliable, and it stays live after graduation. A .nojekyll file in the repository root tells GitHub Pages to skip the Jekyll build and serve the files directly. The result is a fast, dependency-free static site with no ongoing cost.
